CREATE TABLE IF NOT EXISTS weekly_issues (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    domain_id   TEXT NOT NULL REFERENCES domains(id),
    year        INTEGER NOT NULL,
    week        INTEGER NOT NULL,           -- ISO 8601 周数 (1-53)
    start_date  DATE NOT NULL,              -- 该周周一
    end_date    DATE NOT NULL,              -- 该周周日
    title       TEXT,                        -- 例 "第15周 AI周报：Agent安全危机"

    full_md     TEXT,                        -- 渲染后完整 markdown
    daily_issue_ids TEXT,                    -- JSON array: 本周包含的 issue IDs
    status      TEXT NOT NULL DEFAULT 'draft',  -- draft | published
    generated_at TIMESTAMP,
    published_at TIMESTAMP,
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(domain_id, year, week)
);

CREATE INDEX IF NOT EXISTS idx_weekly_domain_year_week

## 三、运维稳定性

### 3.1 gate warn 自动重试 (H3 + L2)

**目标行为变更**:

| 状态 | v1.0.0 行为 | v1.0.1 行为 |
|------|------------|------------|
| gate pass | 推 test + prod | 不变 |
| gate warn | 推 test，不推 prod，exit 0 | **不推任何频道，exit 1 → 触发 retry** |
| gate fail | 推 test (alert)，不推 prod，exit 1 | 不变 |
| retry 第 1-2 次 warn | (无) | 不推任何频道，exit 1 → 继续 retry |
| retry 第 3 次仍 warn | (无) | **降级：视为 pass，推 test + prod** |

**run.go 改动**（~30 行）:

```go
// 在 runPipeline() 入口读取 retry count
retryCount, _ := strconv.Atoi(os.Getenv("BRIEFING_RETRY_COUNT"))

// gate check 后（现 L642）
if !report.Pass && report.Warn {
    if retryCount < 3 {
        // 不推任何频道，exit 1 触发 systemd retry
        return fmt.Errorf("gate warn (retry %d/3): %s", retryCount, strings.Join(report.Warnings, "; "))
    }
    // 第 3 次仍 warn → 降级为 pass，正常推送
    stage(fmt.Sprintf("gate warn: retry %d exhausted, treating as pass", retryCount))
    report.Pass = true
    report.Warn = false
}
```

**新 systemd 文件**:

`/etc/systemd/system/briefing-daily-retry@.service`:
```ini
[Unit]
Description=briefing-v3 daily retry (attempt %i)

[Service]
Type=oneshot
User=root
WorkingDirectory=/root/briefing-v3
EnvironmentFile=/root/briefing-v3/config/secrets.env
Environment=BRIEFING_RETRY_COUNT=%i
ExecStart=/root/briefing-v3/bin/briefing run --target auto
StandardOutput=journal
StandardError=journal
TimeoutStartSec=900
```

**修改 `briefing-daily.service`**:
```ini
OnFailure=briefing-daily-retry-timer.service
```

**`/etc/systemd/system/briefing-daily-retry-timer.service`**:
```ini
[Unit]
Description=briefing-v3 retry scheduling

[Service]
Type=oneshot
ExecStart=/bin/bash -c '\
  current=$(systemctl show briefing-daily.service -p MainPID --value 2>/dev/null); \
  count=1; \
  if systemctl is-failed briefing-daily-retry@1.service 2>/dev/null; then count=2; fi; \
  if systemctl is-failed briefing-daily-retry@2.service 2>/dev/null; then count=3; fi; \
  systemd-run --on-active=30m --unit=briefing-daily-retry@${count}.service'
```

实际上这有点复杂。简化方案——直接用 3 个固定 timer:

**简化版**: 在 `briefing-daily.service` 的 `OnFailure` 中链 `briefing-daily-retry@1.timer`，retry@1 的 `OnFailure` 链 retry@2.timer，retry@2 链 retry@3.timer。每个 timer 都 `OnActiveSec=30min`。

```ini
# briefing-daily-retry@.timer
[Unit]
Description=briefing-v3 retry timer (30min delay)

[Timer]
OnActiveSec=30min
Unit=briefing-daily-retry@%i.service
```

```ini
# briefing-daily-retry@.service
[Unit]
Description=briefing-v3 retry attempt %i
OnFailure=briefing-daily-retry@%I-next.service  # 不行，需要递增

[Service]
Type=oneshot
User=root
WorkingDirectory=/root/briefing-v3
EnvironmentFile=/root/briefing-v3/config/secrets.env
Environment=BRIEFING_RETRY_COUNT=%i
ExecStart=/root/briefing-v3/bin/briefing run --target auto
StandardOutput=journal
StandardError=journal
TimeoutStartSec=900
```

**最终简化方案**: 不用 systemd 模板实例化，用 3 个显式 service+timer 文件对。虽然文件多了但逻辑清晰，不用 shell 脚本算 count。

```
briefing-daily.service         → OnFailure=briefing-retry-1.timer
briefing-retry-1.timer         → OnActiveSec=30min → briefing-retry-1.service
briefing-retry-1.service       → RETRY_COUNT=1 → OnFailure=briefing-retry-2.timer
briefing-retry-2.timer         → OnActiveSec=30min → briefing-retry-2.service
briefing-retry-2.service       → RETRY_COUNT=2 → OnFailure=briefing-retry-3.timer
briefing-retry-3.timer         → OnActiveSec=30min → briefing-retry-3.service
briefing-retry-3.service       → RETRY_COUNT=3 (不再链 OnFailure，失败就是失败)
```

共 6 个新 systemd 文件 + 改 `briefing-daily.service` 的 OnFailure 一行。

**验证确认的注意事项**:
- 每个 retry service 必须自带 `EnvironmentFile=` 和 `WorkingDirectory=`（不从父 service 继承）
- retry service **不应**运行 `ExecStartPre=briefing-daily-reset-dedup.sh`（否则清空当天去重记录）
- retry service **应该**运行 `ExecStartPre=cloudflared-hostname-sync.sh`（防 hostname 漂移）
- 每个 retry service 需设 `TimeoutStartSec=900`（与主 service 一致）

## 七、风险与边界情况

| 风险 | 应对 |
|------|------|
| 跨年周（01-01 属于上年 W53） | `time.ISOWeek()` 返回正确的 isoYear，路径用 isoYear 不用 calendar year |
| 周中只有部分天有日报（刚上线/服务中断） | 允许，周报 prompt 明确说"本周共 N 天日报"，有多少写多少 |
| 周报 LLM 不返回 JSON | 复用 infocard 的 fenced JSON 提取逻辑 |
| updateHomepage glob 到 blog/ 下的 md | glob pattern 排除 `blog/` 目录 |
| Hextra sidebar 共享问题 | 周报用 blog 类型（无 sidebar），天然隔离，已验证 |
| retry 链死循环 | retry@3 不设 OnFailure，最多 3 次 |
| 日报 URL 包含周报反向链接但周报还没生成 | 链接固定可预测（/blog/weekly/YYYY-Www/），Hugo 渲染 404 不影响页面，等周报生成后自动可达 |
