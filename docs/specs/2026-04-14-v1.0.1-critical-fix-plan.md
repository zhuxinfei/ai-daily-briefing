## 3. 核心设计原则（6 条）

1. **文字必补 / 图片可选**（2026-04-14 用户明确，升级版）：
   - **所有文字内容必须齐全**（5 个 section + insight 行业洞察 + insight 启发 + summary 3 句摘要），任一缺失 → **想办法补**（重试 / 换 prompt / 换模型 / repair / 人工介入），**禁止 fallback 兜底文案替代**，**禁止降级推送**
   - **图片/图表尽可能配上**（headline PNG + 12 张 infocard PNG + mermaid 关系图）—— 这是**可选**项，没配上也能正常推，不叫"降级"
   - **"零降级"是底线**：不接受 buildFallbackSummary 这类用默认文案替代真内容的行为（v1.0.0 summary fail-soft 的老做法被撤回）

2. **局部失败必须补，不能放弃**（2026-04-14 用户原话"局部缺信息应该想办法补，不是一再降级"）— 救援链：
   ```
   Attempt 1 (08:00) 失败
     → orchestrator 根据 D4 公式重试（08:25 / 08:45 / 09:05）
     → 任一次齐全 = 推送 = 完结
   全部失败
     → 09:05 最后一次前，自动触发 briefing repair --section <缺失的>
         只补失败 section（不重跑整条），用剩下的时间重试 LLM
     → repair 成功 → 推送
     → 09:25 仍未齐 → 发**紧急告警**到测试频道（附缺失 section、LLM error、下一步建议），让人工介入
   ```
   即：**没有"静默放弃不发早报"这种结局**，要么推齐全版，要么响铃让人知道。

3. **Fail-loud 优先（for 文字）** — 严禁推**文字降级**内容到 prod（今早 04-14 08:10 的状况不可再现）

4. **状态机就是信号源** — gate 不再算 `failedSections`，而是查 DB `status='validated' section 数`

5. **调试模式显式分离** — `BRIEFING_MODE=debug|prod` 环境变量，debug 模式无论 gate 什么结果都不推 prod

6. **重跑而非重头跑** — LLM 502 重试策略改为"分钟级退避"；失败后的**单 section 级重跑**用 `briefing repair --section opensource` 不动别的 section

| 产物 | 失败时 | gate 行为 | "补"的手段 |

| insight 行业洞察 | **必须补齐** | hard fail 不推 | insight stage 重试 → repair insight |

## 4. 已定决策（5 个）

### D1: exit code 映射

| code | 含义 | orchestrator 行为 |
|---|---|---|
| 0 | success | 不重试 |
| 2 | transient (LLM 5xx / timeout / connection reset) | 重试 |
| 3 | content (gate hard fail, 某 section 经 repair 仍缺) | 重试→仍失败→exit 6 |
| 4 | config/auth (secrets 缺失, API key 无效, endpoint unreachable) | 不重试 + 告警 |
| 5 | infra (DB lock, disk full, panic) | 不重试 + 告警 |
| **6** | **needs_human**（09:25 之后仍有 section 缺失，已发告警）| 不重试，systemd 不 restart，等人工 `briefing repair` |

### D2: BRIEFING_MODE

- `secrets.env` 必填，缺失则 `briefing run` 直接 exit 4 "missing BRIEFING_MODE"
- 提供 `config/secrets.env.example` 默认 `BRIEFING_MODE=debug`
- `BRIEFING_MODE=debug` → 无论 gate 结果，publish 只推 `SLACK_TEST_WEBHOOK`
- `BRIEFING_MODE=prod` → 按 gate 决定（pass → prod + test，warn → test only，fail → 告警）

### D3: 历史脏数据处理

- **不主动修**数据库历史记录
- orchestrator pre-flight 做**自愈**：
  ```bash
  # 若今天 issue 已推 prod 但 status='generated'，自动补写
  UPDATE issues SET status='published',
         published_at = COALESCE(published_at, CURRENT_TIMESTAMP)
  WHERE issue_date=today
    AND status='generated'
    AND EXISTS(SELECT 1 FROM deliveries
               WHERE issue_id=issues.id
                 AND channel='slack_prod' AND status='sent')
  ```
- 同时修 Bug K（ensureIssue 保护 status 不被降级覆盖）

### D4: orchestrator 重试公式

- 目标：08:00 开始 → 09:30 硬截止 = **90 分钟总预算**
- 最多 4 次尝试，每次留 10 分钟 safety margin
- 固定时刻 schedule（实现简单）：
  - T=08:00 第 1 次
  - T=08:25 第 2 次（若上次 exit=2 且状态可恢复）
  - T=08:45 第 3 次
  - T=09:05 第 4 次（最后机会）
  - T=09:25 告警放弃 + orchestrator exit 非 0
- 若任一次 exit=3/4/5 → 立即告警 + 放弃（不再重试）
- orchestrator 短路：启动时查 DB `issue.status='published'` → 立即 exit 0

### D5: validator regenerate

- 位置：**`summarize.go:97` 的 retry for-loop 内**（与 LLM 502 重试同一层）
- 逻辑：
  ```go
  raw, err := g.chatComplete(...)
  if err != nil { retry }
  if !ValidateCompose(raw).OK { retry }  // 新加一行
  return cleaned
  ```
- 理由：单一重试心智模型，不增加嵌套层
